## 対象読者

Claude Code を数日〜数週間触って、次のような壁を感じている人。

- 毎回プロジェクトの前提を説明し直すのが面倒
- 危険なコマンドを勝手に実行してほしくない
- 外部サービス（GitHub / Slack / DB / ブラウザ）を Claude から直接操作したい
- 長いタスクを複数の Claude に分業させたい

## 1. CLAUDE.md：プロジェクト定数の外部化

プロジェクトルートに `CLAUDE.md` を置いておくと、Claude Code が起動時に自動で読み込みます。ここに以下を書いておくと、毎回の説明を消せます。

- 技術スタック（フレームワーク・言語・バージョン）
- コーディング規約
- 禁止事項
- 環境前提（Windows の `/dev/null` vs `NUL` など）
- よく使うコマンド

親ディレクトリにも置けるので、**共通ルールは親に、プロジェクト固有は子に** 分けると管理しやすい。

## 2. Hooks：ツール呼び出しの前後にシェルを挟む

Claude Code には、ツール実行前後に任意のシェルコマンドを走らせるフック機構があります。

| 種類 | タイミング | よくある用途 |
|---|---|---|
| `PreToolUse` | ツール呼び出しの直前 | 危険コマンドのブロック、バリデーション |
| `PostToolUse` | ツール実行後 | フォーマッタ、リンター、テスト実行 |
| `UserPromptSubmit` | ユーザー発言の送信前 | リマインダ、追加コンテキスト挿入 |

実用的な最初の一歩は `PostToolUse` で `Edit` の後にフォーマッタを走らせること。Claudeの編集結果が常にフォーマット済みで返ってくるようになります。

:::message alert
フックはローカルシェルで実行されます。**安全でないコマンドを登録すると取り返しがつきません**。他人の設定を中身を読まずにコピーしないこと。
:::

## 3. MCP：外部サービスへの標準プロトコル

MCP（Model Context Protocol）は、Claude から外部サービスへアクセスするための標準化プロトコル。MCP サーバーを起動しておくと、Claude Code がその機能を「ツール」として呼び出せます。

代表的なもの:

- GitHub MCP（Issue/PR 操作）
- Slack MCP（メッセージ送信）
- **Playwright MCP**（ブラウザ操作）← Web 開発なら最優先
- Postgres / SQLite MCP（DB 直クエリ）
- Filesystem MCP（プロジェクト外のファイルアクセス）

Playwright MCP が入ると、こういう指示が通るようになります。

> `localhost:3000` を開いて `/blog/claude-code-guide` にアクセスして、スクリーンショットを撮って。404 なら対応するファイルを探して原因を特定して。

従来は人間が挟まないといけなかったブラウザ検証の往復が、Claude 側で完結します。

## 4. Sub-agent：並列分業

Claude Code は、別の Claude インスタンスにサブタスクを投げられます。メインが指揮、サブが個別作業。並列化と、コンテキスト消費の分離に効きます。

使いどころ:

- 大量の調査系（5案件のステータス確認など）
- 影響範囲が閉じた実装（ヘッダーコンポーネントの作成など）
- 大量ファイル読み込みを伴う作業（メインの会話ログを汚したくないとき）

**注意**: サブエージェントは呼び出し時の指示だけを持って動きます。メインで共有した前提は引き継がれないので、**自己完結したプロンプト**（何を、何のために、どのゴールで）を書く必要があります。

## 5. 権限設定：事故を防ぐ最後の砦

Claude Code を「任せきり」にするのは危険です。次の操作は確認を挟む設定にしておくべき。

- `git push` / `rm -rf` などの破壊的操作
- 本番 DB への書き込み
- `.env` / credentials へのアクセス
- 外部 API への有料リクエスト

個人の実験プロジェクトは自動実行多め、共有/本番リポジトリは確認多めにする段階設計がおすすめ。

## 実運用のコツ

1. **`CLAUDE.md` を育てる** — 同じ説明を2回したら書き足す
2. **フックは最小構成から** — 最初は PostToolUse のフォーマッタ1つだけ
3. **MCP は自分のワークフロー起点で** — 流行りで選ばず、今週一番時間を食っている作業を起点に
4. **サブエージェントは閉じた作業に限定** — 入力と出力がはっきりしている作業だけに使う
